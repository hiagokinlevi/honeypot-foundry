# SIEM Alerting Rules

This document provides preconfigured, production-ready detection rules for honeypot telemetry generated by `honeypot-foundry`.

Rules are designed to detect:
- Brute force attempts
- Malware upload behavior
- Command execution attempts
- Lateral movement behavior

> These rules assume normalized JSON events from honeypot services (SSH, HTTP, API, FTP, RDP) and are intentionally SIEM-agnostic. Example mappings for Splunk and Elastic are included.

---

## Event Field Assumptions

Use the following common fields (or equivalent mapped fields):

- `timestamp` (ISO8601)
- `event_type` (e.g., `auth_attempt`, `http_request`, `ftp_command`, `rdp_probe`)
- `service` (`ssh`, `http`, `api`, `ftp`, `rdp`)
- `source.ip`
- `source.port`
- `destination.ip`
- `destination.port`
- `http.method`
- `http.path`
- `http.user_agent`
- `http.status_code`
- `auth.username_masked`
- `auth.password_masked`
- `auth.success` (expected false for decoys)
- `payload` (raw request metadata/body excerpt where configured)
- `session.id`

If your pipeline uses different names, create a simple field mapping in your SIEM parser.

---

## Severity Model

- **Low**: Initial recon/probe
- **Medium**: Confirmed suspicious behavior (single vector)
- **High**: Strong attack pattern (repetition or dangerous technique)
- **Critical**: Multi-stage behavior indicating active compromise workflow

---

## Rule 1 — SSH/FTP Brute Force Attempts

**Goal:** Detect repeated credential attempts from a source against SSH or FTP honeypots.

**Logic:**
- Group by `source.ip`, `service`
- Count failed auth attempts in rolling window
- Trigger on threshold

**Recommended threshold:**
- `>= 10` failed attempts in `5 minutes` (Medium)
- `>= 25` failed attempts in `10 minutes` (High)

**Pseudo query:**

```text
WHERE event_type IN ("auth_attempt", "login_attempt")
  AND service IN ("ssh", "ftp", "api")
  AND auth.success = false
GROUP BY source.ip, service, WINDOW(5m)
HAVING COUNT(*) >= 10
```

**Splunk example (SPL):**

```spl
index=honeypot (event_type=auth_attempt OR event_type=login_attempt) service IN (ssh,ftp,api) auth.success=false
| bin _time span=5m
| stats count as attempts values(auth.username_masked) as usernames by _time, source.ip, service
| where attempts >= 10
```

**Elastic example (KQL + threshold rule):**

```kql
event_type:("auth_attempt" or "login_attempt") and service:("ssh" or "ftp" or "api") and auth.success:false
```

Then configure threshold:
- Group by: `source.ip`, `service`
- Threshold: `10`
- Time window: `5m`

## Rule 2 — Malware Upload Attempt (HTTP/FTP)

**Goal:** Detect attempted delivery of executable/script payloads.

**Indicators:**
- Suspicious upload endpoints (`/upload`, `/file`, `/admin/upload`, `/api/import`)
- Suspicious file extensions in path/body (`.exe`, `.dll`, `.ps1`, `.bat`, `.vbs`, `.js`, `.jar`, `.elf`, `.php`)
- Multipart upload with executable MIME patterns
- FTP `STOR` command to suspicious filename

**Pseudo query:**

```text
WHERE (
  service = "http"
  AND event_type = "http_request"
  AND (
    http.method IN ("POST", "PUT")
    AND (
      http.path MATCHES "(?i).*(upload|import|file|attachment).*"
      OR payload MATCHES "(?i)\\.(exe|dll|ps1|bat|vbs|js|jar|elf|php)"
      OR payload MATCHES "(?i)(application/x-msdownload|application/octet-stream|multipart/form-data)"
    )
  )
)
OR (
  service = "ftp"
  AND event_type = "ftp_command"
  AND payload MATCHES "(?i)^STOR\\s+.*\\.(exe|dll|ps1|bat|vbs|js|jar|elf|php)$"
)
```

**Severity:** High

**Tuning tips:**
- Allowlist known internal scanner IPs used for purple-team exercises.
- Suppress repeated hits from same `source.ip` + filename hash/signature for 30m.

## Rule 3 — Command Execution Attempt

**Goal:** Detect attacker attempts to execute shell/system commands through web/API parameters or protocol commands.

**Indicators (examples):**
- Shell metacharacters: `;`, `&&`, `||`, `` ` `` , `$(`
- Known command strings: `whoami`, `id`, `uname`, `cat /etc/passwd`, `powershell`, `cmd.exe`, `curl`, `wget`, `nc`, `bash -i`
- Encoded command delivery attempts (`base64`, `-enc`, `FromBase64String`)

**Pseudo query:**

```text
WHERE service IN ("http", "api", "ftp", "ssh")
  AND (
    payload MATCHES "(?i)(;|&&|\\|\\||`|\\$\\()"
    OR payload MATCHES "(?i)\\b(whoami|id|uname|ifconfig|ipconfig|cat\\s+/etc/passwd|powershell|cmd\\.exe|wget|curl|nc|bash\\s+-i)\\b"
    OR payload MATCHES "(?i)(base64|frombase64string|-enc\\s+)"
  )
```

**Severity:** High

**Escalation to Critical:**
- Same `source.ip` triggers this rule and malware upload rule within 15 minutes.

## Rule 4 — Lateral Movement Behavior

**Goal:** Detect scanning/probing across multiple decoy services or hosts from the same source.

**Behavioral pattern:**
- Same `source.ip` touches multiple services (`ssh`, `rdp`, `ftp`, `http`, `api`) in short period
- OR accesses many destination ports/hosts indicating internal propagation attempts

**Recommended thresholds:**
- `>= 3` distinct services in `10 minutes` (High)
- `>= 5` distinct destination ports in `10 minutes` (High)
- Both conditions true in `10 minutes` (Critical)

**Pseudo query:**

```text
WHERE event_type IN ("connection", "auth_attempt", "http_request", "rdp_probe", "ftp_command")
GROUP BY source.ip, WINDOW(10m)
HAVING DISTINCT_COUNT(service) >= 3
   OR DISTINCT_COUNT(destination.port) >= 5
```

**Splunk example (SPL):**

```spl
index=honeypot event_type IN (connection,auth_attempt,http_request,rdp_probe,ftp_command)
| bin _time span=10m
| stats dc(service) as svc_count dc(destination.port) as dport_count values(service) as services values(destination.port) as ports by _time, source.ip
| where svc_count >= 3 OR dport_count >= 5
```

## Correlation Rule — Multi-Stage Intrusion Workflow

**Goal:** Raise confidence by correlating multiple detections from same source.

**Trigger when, within 30 minutes, same `source.ip` matches at least 2 of:**
1. Brute force
2. Malware upload
3. Command execution
4. Lateral movement

**Severity:** Critical

**Response guidance:**
- Auto-create incident
- Attach full session timeline
- Enrich with GeoIP, ASN, and threat intel reputation

---

## Operational Guidance

- Start in **alert-only mode** for 7 days to baseline noise.
- Review top talkers and add exercise infrastructure allowlists.
- Keep honeypot detections high-signal: honeypot traffic is inherently suspicious, so thresholds can be lower than production app detections.
- Route Critical alerts to on-call SOC channel immediately.

---

## Minimal Alert Payload Template

Include these fields in SIEM alert outputs:

- `rule_name`
- `severity`
- `source.ip`
- `service`
- `event_count`
- `first_seen`
- `last_seen`
- `sample_paths_or_commands`
- `session_ids`

This ensures consistent triage across all four detection categories.
